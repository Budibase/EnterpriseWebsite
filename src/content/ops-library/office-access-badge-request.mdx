<h2 id="overview">Overview</h2>

Office Access Badge Request turns a recurring internal workflow into an agent-led operation. A Budibase agent captures the request, checks the documented policy, uses live system data where needed, and either resolves the work automatically or hands it to the right person with a clear audit trail.

**Request**
A requester submits a office access badge request case through Slack or a Budibase form.

**Agent Triage**
The agent validates the request details, checks Sharepoint guidance, and gathers the live system context needed to decide the next step.

**Agent Resolution**
Documented low-risk cases are completed automatically and written back to the request queue with a clear rationale.

**Escalation**
Requests that require approval, exception handling, or additional judgement are routed to the appropriate owner.

**Human Resolution**
The owning team reviews the case, records the final decision, and closes the request.

<h2 id="agents">Agents</h2>

This operation uses one Budibase agent to receive requests for office access badge request, gather the required context, update the operational records, and decide whether the work can be completed automatically or needs human review.

```text
# Role
You are the Office Access Badge Request operation agent for HR / Admin.

# Goal
Move each request from intake to outcome using workspace data, connected tools, and the approved guidance stored in Sharepoint.

# Inputs
- `request_id` - the unique request identifier
- `requester` - the employee or operator who submitted the request
- `submitted_channel` - Slack or the Budibase intake form
- `employee_name` - Employee Name provided as part of the request.
- `office_location` - Office Location provided as part of the request.
- `badge_type` - Badge Type provided as part of the request.

# Workflow
1. Confirm that the request belongs to the Office Access Badge Request operation and ask follow-up questions in Slack if required details are missing.
2. Consult Sharepoint before making any policy decision, recommendation, or exception judgment.
3. Keep the request record in sync with the latest status using {{ postgresql.office_access_badge_request_queue.update_row }}.
4. Auto-resolution: if the request clearly matches a documented low-risk path, complete the allowed system action using {{ serviceNow.create_ticket }} and {{ serviceNow.update_ticket }}.
5. Escalation: if approval is required, the request is ambiguous, or the policy does not allow autonomous completion, update the request to `Escalated` and write an audit note with {{ postgresql.office_access_badge_request_audit_log.create_row }}.
6. Human resolution: once a human decision is recorded, update the request status, summarize the outcome, and close the loop with the requester.

# Output
Return a short operational summary that includes the request status, the next owner, and the reason for the current decision.

# Rules
- Do not invent policy. Use Sharepoint as the source of truth for approval and exception handling.
- Do not perform actions outside the documented operation scope.
- Keep all updates concise, auditable, and aligned to the request record.
```

<h2 id="data">Data</h2>

### Tables

**Office Access Badge Request Queue**: Primary queue for office access badge request work.

- `request_id` : Text - Unique identifier for the request.
- `requester` : Single User - The employee or operator who submitted the request.
- `submitted_at` : Date/Time - Timestamp when the request was created.
- `status` : Single Select - Current lifecycle state such as Submitted, In review, Escalated, or Closed.
- `priority` : Single Select - Operational urgency used to order the queue.
- `employee_name` : Text - Employee Name captured for this operation.
- `office_location` : Text - Office Location captured for this operation.
- `badge_type` : Text - Badge Type captured for this operation.
- `agent_summary` : Long Form Text - The latest agent-generated summary of the request and next action.
- `decision` : Single Select - Approved, Rejected, Auto-resolved, Escalated, or Waiting for input.
- `decision_owner` : Text - The human or team currently responsible for the next action.

```csv
request_id,requester,submitted_at,status,priority,employee_name,office_location,badge_type,agent_summary,decision,decision_owner
OFFICE-A-1041,alex.chen@company.com,2026-05-15T09:30:00Z,Escalated,Medium,Alex Chen,London HQ,Employee,Agent validated the request and prepared the next action for office access badge request.,Escalated,Operations owner
```

**Office Access Badge Request Audit Log**: Event log for every meaningful action taken during office access badge request.

- `event_id` : Text - Unique identifier for a office access badge request audit event.
- `request_id` : Text - Reference to the parent request record.
- `event_time` : Date/Time - Timestamp for the recorded event.
- `actor` : Text - The agent, approver, or system that performed the action.
- `event_type` : Text - Short label for the action that occurred.
- `notes` : Long Form Text - Plain-language explanation of the action or decision.

```csv
event_id,request_id,event_time,actor,event_type,notes
OFFICE-A-EVT-1,OFFICE-A-1041,2026-05-15T10:05:00Z,operation-agent,Escalated,Audit note for office access badge request.
```

**Slack**

- Receives the initial request and any follow-up questions.
- Sends status updates, approvals, and completion messages back to the requester or owning team.

**ServiceNow**

- Creates and updates fulfilment or incident tickets when work needs structured handoff.
- Keeps downstream IT tasks aligned with the status of the Budibase request.

**PostgreSQL**

- Stores the request state, timestamps, and operational metadata for the workflow.
- Keeps a durable audit trail of agent and human actions from intake to closure.

**Sharepoint**

- Provides the policy documents, HR or finance guidance, and intake templates the agent consults before acting.

### Adding Knowledge

The agent uses Sharepoint as its knowledge source for policy, runbooks, and exception criteria. This matters because the operation depends on documented approval rules instead of improvised judgement, and the agent can retrieve the relevant procedure, ownership guidance, and escalation criteria at the point of decision.

### Office Access Badge Request Workspace

A lightweight Budibase app used by the owning team to review, approve, and track office access badge request work from one queue.

**Request Intake Form**: Captures the structured details needed to open a office access badge request case when chat alone is not enough.

**Office Access Badge Request Queue**: Shows open requests, priority, status, and who owns the next action.

**Request Detail**: Displays the full request history, agent summary, audit events, and any related approval notes.

<h2 id="automations">Automations</h2>

**Create request record**: On Demand - Creates or updates the request row when structured details are captured in the app or chat flow.

**Notify approver**: On Update Row - Sends the right approval or exception notification when the request enters an escalated state.

**Close request**: On Update Row - Posts the final outcome, timestamps the request, and appends a closing audit event.
